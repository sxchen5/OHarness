# Generator Prompt 模板 — 任务执行

## 使用方式

根据当前任务类型，选择对应的模板发送给 Claude Code。每个模板都会自动注入约束层和信息层上下文。

**核心原则：测试驱动开发（TDD）+ 实时性预算优先**

所有含运行时逻辑的任务（协议栈函数 / 嵌入式驱动 / 数值算法）必须遵循 TDD 流程，且 Red 阶段必须把 cycle 计数断言写进单测：

1. **Red** — 先写测试（GTest / cmocka），覆盖正负边界 + 实时性预算 cycle 计数断言，运行确认 FAIL
2. **Green** — 写最小实现，编译通过 + 单测全部 PASS（含 cycle 断言）
3. **Refactor** — 在不破坏 cycle 预算的前提下重构，单测仍 PASS

不含运行时逻辑的任务（头文件 / 数据结构定义 / 协议事实查询）不要求 TDD，但仍需编译过 / 静态检查过 / 引用段落可溯源。

```
3. 内存：禁止 hot path malloc / free，预分配 + 对象池
4. 中断 / RT 安全：函数若运行在中断 / RT 线程上下文，禁止持锁 / 阻塞 IO / printf
5. 编译：`make` 干净通过，`-Wall -Wextra -Werror` 零告警
6. 运行 `make test`，确认全部 PASS（含 cycle 断言）

1. 消除重复 / 提炼公共宏 / inline 内联热点
2. 重构后重跑单测，**cycle p95 必须仍 ≤ BUDGET**；若回归 ≥ 5%，回退或换实现
```

## 模板2: 嵌入式驱动（TDD）

**适用**：CPRI 控制字组帧 / FPGA 寄存器读写 / 中断处理 / DMA 收发 / SPI / I2C / GPIO 时序控制

```
{通用前缀}

### 任务: {任务ID} - 实现 {驱动名}

**文件路径**: `{驱动源文件，如 bbu/hal/cpri/cpri_ctrlword.c}`
**头文件**: `{驱动头文件}`
**硬件接口契约**: `{接口文档路径或编号，如 docs/hw-interface/cpri-v0.9.md §3.2}`
**寄存器地图**: `{寄存器表，如 docs/fpga/regs.csv}`
**实时性预算**: {单次调用 / 中断响应 cycle 上限}

**TDD 流程**:

**第1步 — Red（先写单测 + cycle 断言）**:
1. 创建测试文件: `{测试路径}`，框架 GTest / cmocka
2. 寄存器/外设打桩：用 mock MMIO 层（替换 `readl/writel` 为 fake，记录访问序列）
3. 用例覆盖：
   - 正向：组帧/拆帧字节序、控制字字段位偏移与硬件契约**逐位对齐**
   - 负向：非法寄存器地址 / 越权写只读位 / 时序约束违反 → 返回错误码
   - 边界：DMA 描述符环回绕 / 缓冲区对齐 / cache line 边界
4. **cycle 断言（必须）**：
   - 中断处理 / 关中断段：实测关中断时长 ≤ 预算
   - 寄存器读写热路径：循环 N=1000 次取均值，p95 ≤ BUDGET
5. 运行 `make test`，确认 FAIL

**第2步 — Green（最小实现）**:
1. 严格按硬件接口契约写位段宏（`#define FIELD_X_SHIFT / _MASK`），禁止裸魔法数
2. 内存屏障 / cache 一致性：DMA 收到的数据 `dma_invalidate` 后再读；DMA 发送前 `dma_flush`
3. 中断安全：ISR 内最小化（只置 flag / 唤醒下半部），禁止打印 / 锁
4. 寄存器写后读回校验（关键控制字必须有 readback 断言）
5. 编译 `-Wall -Wextra -Werror` 零告警；静态分析（cppcheck / clang-tidy）零 critical
6. 运行 `make test`，确认 PASS

**第3步 — Refactor（不破坏 cycle 预算 + 中断安全）**:
1. 不放宽 ISR 时长上限；不增加锁层级
2. 重跑 cycle 断言，p95 不回归
3. 寄存器位段命名与硬件文档保持一致

**交付物**:
- 驱动: `{源文件}` + `{头文件}`
- 测试: `{测试文件}`（含 mock MMIO 访问序列回放）
- readback / cycle 实测数据
- 硬件接口契约引用清单
```

## 执行流程

对每个任务：

1. **选择模板**: 协议栈函数 / 嵌入式驱动 / 数值算法 / 协议事实查询 四选一
2. **注入上下文**: 填入任务ID、文件路径、协议条款引用、cycle 预算、依赖关系
3. **TDD 执行**: 先写测试（含 cycle 断言）→ 最小实现 → 在不破坏预算前提下重构
4. **提交验证**: 用 Evaluator 检查（见 evaluator.md），实时性预算证据必须随交付
5. **记录度量**: cycle p50/p95/p99 实测、协议引用清单、硬件接口契约 readback 结果（见 metrics.md）
